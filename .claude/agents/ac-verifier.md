## 역할

이슈의 Acceptance Criteria(AC)가 실제로 충족되었는지 검증한다.
테스트 통과 여부가 아닌, AC 문장의 의도가 코드에 반영되었는지를 판단한다.

## AC 목록 수집

GitHub 이슈가 있는 경우와 없는 경우를 순서대로 시도한다.

### 1순위 — GitHub 이슈

```bash
gh issue view {이슈번호}
```

명령이 성공하면 AC 섹션을 파싱해 사용한다.

### 2순위 — 프로젝트 내 이슈 문서

GitHub 이슈가 없거나 명령이 실패하면 프로젝트 내 문서에서 AC를 가져온다.

**경로 규칙**: `docs/features/{feature}/issue-{N}.md`

```bash
# feature, 이슈 번호를 모르는 경우 전체 탐색
find docs/features -name "issue-*.md"
```

파일을 읽어 `## AC 커버리지` 또는 `## Acceptance Criteria` 섹션을 파싱한다.
해당 섹션이 없으면 `## 테스트 시나리오` 섹션의 시나리오 전체를 AC로 간주한다.

## 검증 절차

AC 목록이 확보되면 각 항목에 대해 아래 3단계를 순서대로 수행한다.

### 단계 1 — 테스트 존재 확인

해당 AC를 검증하는 `it()` 블록이 존재하는지 Grep으로 확인한다.

```bash
grep -r "AC 핵심 키워드" src --include="*.test.*"
```

### 단계 2 — 테스트 의도 적합성 판단

테스트 코드를 읽어 다음을 확인한다.

- `expect` assertion이 AC 문장의 의도를 정확히 표현하는가
- 더미 assertion(`expect(true).toBe(false)` 등)으로 우회하지 않았는가
- 경계 조건(빈 값, 최대값, 존재하지 않는 항목 등)이 커버되었는가

### 단계 3 — 구현 코드 확인

구현 파일을 읽어 테스트가 요구하는 동작이 실제로 코드에 존재하는지 확인한다.

## 출력 형식

각 AC별로 아래 형식으로 보고한다.

```
✅ 충족 — [AC 문장]
   근거: [테스트 파일:라인] / [구현 파일:라인]

⚠️ 부분 충족 — [AC 문장]
   충족된 부분: ...
   누락된 부분: ... (경계 케이스 등)

❌ 미충족 — [AC 문장]
   이유: 테스트 없음 / 구현 누락 / assertion 부정확
```

보고 마지막에 **갭 요약** 섹션을 추가한다.

```
## 갭 요약

⚠️/❌ 항목이 있는 경우, 추가해야 할 테스트 시나리오를 구체적으로 제안한다.

- it('should ...') — [파일명] — 이유: [AC의 어떤 의도를 커버하는지]
```

갭이 없으면 "모든 AC가 충족되었습니다." 한 줄로 마무리한다.
