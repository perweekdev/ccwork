`$ARGUMENTS`에서 이슈 번호를 파싱해, 타입 오류·보안 취약점·환경변수 노출 여부를 점검하고 결과를 분류한다.

## 시작 전

### 인자 파싱

`$ARGUMENTS`를 이슈 번호(`{issue}`)로 사용한다.

`{issue}`가 없으면: "점검할 이슈 번호를 알려주세요."

---

## 단계 1: 타입 오류 수집

```bash
npx tsc --noEmit 2>&1
```

오류가 없으면 "타입 오류 없음"으로 기록한다. 오류가 있으면 파일명·줄 번호·오류 메시지를 목록으로 수집한다.

---

## 단계 2: 보안 취약점 수집

```bash
npm audit 2>&1
```

취약점이 없으면 "취약점 없음"으로 기록한다. 있으면 패키지명·severity·설명을 목록으로 수집한다.

---

## 단계 3: 환경변수 노출 확인

`src/` 디렉토리에서 아래 패턴을 검색한다:

- `process.env` 직접 참조
- `.env` 파일에서 읽어온 값을 문자열 리터럴로 하드코딩한 패턴 (`VITE_`, `REACT_APP_` 접두사 포함)

발견된 위치를 파일명·줄 번호와 함께 수집한다.

---

## 단계 4: 결과 분류 및 보고

수집된 전체 결과를 아래 세 등급으로 분류해 보고한다.

| 등급               | 기준                                                              |
| ------------------ | ----------------------------------------------------------------- |
| **즉시 수정 필요** | 보안 위험(high/critical severity) 또는 빌드 실패 수준의 타입 오류 |
| **권장 수정**      | moderate severity 취약점, 코드 품질을 저하시키는 타입 경고        |
| **무시 가능**      | false positive, 의도된 패턴, low/info severity                    |

항목이 없는 등급은 "해당 없음"으로 표시한다.

**"즉시 수정 필요" 목록을 확인한 뒤 수정 진행 여부를 개발자에게 승인받는다.**

---

## 단계 5: 승인 항목 처리

승인된 "즉시 수정 필요" 항목만 처리한다.

- 타입 오류: 해당 소스 파일을 수정한다.
- 취약점: `npm audit fix` (--force 사용 금지)로 처리한다. 자동 수정이 불가한 경우 수동 조치 방법을 개발자에게 안내한다.
- 환경변수 노출: 노출된 값을 환경변수 참조 또는 서버 측 처리로 교체한다.

각 항목 처리 후 단계 1~3을 항목별로 재실행해 수정 결과를 즉시 확인한다.

---

## 단계 6: 재스캔 및 클린 상태 확인

모든 승인 항목 처리 완료 후 단계 1~3 전체를 재실행한다.

"즉시 수정 필요" 항목이 모두 해소되었으면 클린 상태로 보고한다. 잔여 항목이 있으면 사유와 함께 개발자에게 알린다.

---

## 최종 요약 형식

```
이슈: {issue}

타입 오류: {오류 수} / 처리: {처리 수}
보안 취약점: {취약점 수} / 처리: {처리 수}
환경변수 노출: {발견 수} / 처리: {처리 수}

잔여 즉시수정 항목: {수} (0이면 클린)
```

---

## 제약

- 판단이 필요한 항목은 반드시 개발자 승인 후 수정한다.
- 테스트 파일(`*.test.ts`, `*.test.tsx`) 수정 금지.
- `npm audit fix --force` 사용 금지.
- `src/` 디렉토리 밖 소스 파일 수정 금지.
