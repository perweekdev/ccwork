GitHub 이슈 `$ARGUMENTS`에 대한 시그니처를 확정하고 테스트 시나리오를 도출한다.

## 시작 전

1. `$ARGUMENTS`가 비어 있으면 GitHub 이슈 번호를 사용자에게 먼저 묻는다.
2. 아래 세 가지를 동시에 읽는다:
   - `gh issue view $ARGUMENTS` — 이슈 제목, 설명, AC 목록
   - `docs/features/tag/prd.md` — 사용자 스토리, ADR, 용어 정의
   - 관련 코드베이스 파일 — 기존 패턴 파악을 위해 아래를 탐색한다:
     - `src/types/note.ts` — 타입 정의 패턴
     - `src/api/notes.ts` — API 함수 시그니처 패턴
     - `src/context/NotesContext.tsx` — Context 노출 함수 패턴
     - `src/components/` — Props 인터페이스 패턴

---

## 단계 1: 시그니처 확정

이슈 범위 안에서 새로 추가되거나 변경되는 함수·컴포넌트·훅의 시그니처를 도출한다.

### 작성 항목

이슈에 해당하는 항목만 작성한다. 해당 없는 항목은 생략한다.

**함수 시그니처**

```ts
// 함수명(파라미터명: 타입, ...): 반환타입
```

**에러 케이스**

```
- 어떤 조건일 때 → 어떤 에러를 던지는가 (또는 조용히 무시하는가)
```

**컴포넌트 Props 타입**

```ts
interface ComponentNameProps {
  propName: type;
}
```

**훅 반환 타입**

```ts
// useHookName(): { 반환값명: 타입, ... }
```

### 준수 규칙

- 구현 코드는 절대 작성하지 않는다. 시그니처(이름·타입)만 작성한다.
- 기존 패턴을 따른다:
  - API 함수: `src/api/notes.ts`의 함수 시그니처 스타일
  - Context 노출 함수: `createNote` / `updateNote` / `deleteNote` 동사 패턴
  - 컴포넌트 Props: 컴포넌트 바로 위에 `interface {Name}Props` 정의
  - 이벤트 핸들러 props: `on` 접두사 (`onSelect`, `onDelete`)
  - 불리언 props: `is` 접두사 (`isCreating`, `isSelected`)
  - 커스텀 훅: `use` 접두사

```
[GATE] 확정된 시그니처를 개발자에게 보여주고, 승인을 받을 때까지 단계 2로 넘어가지 않는다.
수정 요청이 있으면 시그니처를 조정하고 다시 보여준다.
```

## 단계 2: 시그니처를 issue-{N}.md 상단에 기록

승인된 시그니처를 `docs/features/tag/issue-$ARGUMENTS.md`에 저장한다.
파일이 없으면 새로 생성한다.

```markdown
# Issue $ARGUMENTS 시그니처 & 테스트 시나리오

## 시그니처

(승인된 시그니처 내용)

---

## 테스트 시나리오

(단계 3에서 추가)
```

---

## 단계 3: 테스트 시나리오 도출

승인된 시그니처를 기반으로 테스트 시나리오를 도출한다.

### 분류 기준

| 분류     | 기준                                                        |
| -------- | ----------------------------------------------------------- |
| **정상** | 유효한 입력으로 기대 결과가 반환되는 경우                   |
| **경계** | 최솟값, 최댓값, 빈 배열, 빈 문자열 등 경계 조건             |
| **예외** | 유효하지 않은 입력, 중복, 공백 전용 등 거부되어야 하는 경우 |

### 시나리오 형식

```
[정상/경계/예외] 함수명(또는 컴포넌트명) — should [기대 동작] when [조건]
```

### 작성 규칙

- 시나리오당 한 줄. 테스트 코드는 작성하지 않는다.
- 시그니처의 파라미터·반환 타입·에러 케이스를 빠짐없이 시나리오로 변환한다.
- 정상 1개 이상, 경계 1개 이상, 예외(에러 케이스가 있으면) 1개 이상을 포함한다.

---

## 단계 4: AC 커버리지 대조

```bash
gh issue view $ARGUMENTS
```

이슈의 AC 목록을 읽어 시나리오와 대조한다.

- AC 항목마다 커버하는 시나리오가 1개 이상 존재하는지 확인한다.
- 커버되지 않은 AC가 있으면 해당 AC를 위한 시나리오를 추가한다.
- 대조 결과를 표로 정리해 보여준다:

| AC 항목 | 커버 시나리오 | 상태    |
| ------- | ------------- | ------- |
| AC 내용 | 시나리오 이름 | ✅ / ❌ |

---

## 단계 5: 시나리오를 issue-{N}.md 하단에 추가

도출된 시나리오 전체를 `docs/features/tag/issue-$ARGUMENTS.md`의 "테스트 시나리오" 섹션에 기록한다.

```
[GATE] 완성된 시나리오 목록과 AC 커버리지 표를 개발자에게 보여주고, 승인을 받을 때까지 종료하지 않는다.
수정 요청이 있으면 시나리오를 조정하고 다시 보여준다.
```
