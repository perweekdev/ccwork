`$ARGUMENTS` 기능의 PRD를 작성해 `docs/features/$ARGUMENTS/prd.md`에 저장한다.

## 실행 순서

1. `$ARGUMENTS`가 비어 있으면 기능명을 사용자에게 먼저 묻는다.

2. 스펙 파일을 읽는다.
   - `docs/features/$ARGUMENTS/spec-fixed.md` 가 있으면 그것을 사용한다.
   - 없으면 `docs/features/$ARGUMENTS/` 디렉토리 안의 다른 스펙 파일을 찾아 사용한다.
   - 스펙 파일이 없으면 사용자에게 경로를 묻는다.

3. 기준 포맷 확인을 위해 `docs/features/tag/prd.md`를 참조한다.

4. 아래 PRD 구조에 따라 `docs/features/$ARGUMENTS/prd.md`를 작성한다.

## PRD 구조

### 개요

스펙의 핵심 동작을 2–3문장으로 요약한다. 무엇을, 어디서, 어떻게 동작하는지를 포함한다.

### 사용자 스토리

스펙에서 사용자 행동 단위를 추출해 아래 표 형식으로 작성한다.

| #   | 스토리 (`~하는 사용자로서, ~하고 싶다.`) | 인수 조건 (observable한 결과) |
| --- | ---------------------------------------- | ----------------------------- |
| U1  | ...                                      | ...                           |

- 한 스토리는 하나의 관찰 가능한 사용자 행동에 대응한다.
- 인수 조건은 화면·API·상태 변화로 검증 가능하게 작성한다.

### 기술 결정 (ADR)

스펙의 임의 결정·설계 선택지를 ADR 형식으로 변환한다. 결정이 없으면 스펙에서 추론한다.

각 ADR은 아래 항목을 포함한다:

- **Context**: 결정이 필요했던 배경
- **Decision**: 채택한 방식
- **Alternatives**: 검토했지만 채택하지 않은 대안 및 이유
- **Consequences**: 장단점

### Out of Scope

스펙에서 "미구현", "범위 밖", "향후" 로 표시된 항목을 소제목 + 1–2줄 설명으로 나열한다.

### 용어 정의

스펙에서 도메인 용어 또는 기술 용어를 추출해 표 형식으로 정의한다.

## 작성 규칙

- 모든 내용은 한국어로 작성한다.
- 사용자 스토리 주어는 항상 `~하는 사용자로서`로 시작한다.
- ADR의 Context·Decision·Alternatives·Consequences는 생략하지 않는다.
- 인수 조건은 구현 세부사항이 아닌 관찰 가능한 결과로 작성한다.
- 저장 완료 후 파일 경로와 포함된 사용자 스토리 수, ADR 수를 사용자에게 보고한다.
