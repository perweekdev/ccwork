`$ARGUMENTS` 기능의 이슈를 수직 슬라이싱 원칙에 따라 분해해 `docs/features/$ARGUMENTS/issues.md`에 저장한다.

## 실행 순서

1. `$ARGUMENTS`가 비어 있으면 기능명을 사용자에게 먼저 묻는다.

2. PRD를 읽는다.
   - `docs/features/$ARGUMENTS/prd.md`가 있으면 그것을 사용한다.
   - 없으면 사용자에게 경로를 묻는다.

3. 기준 포맷 확인을 위해 `docs/features/tag/issues.md`를 참조한다.

4. 아래 수직 슬라이싱 원칙과 이슈 구조에 따라 `docs/features/$ARGUMENTS/issues.md`를 작성한다.

## 수직 슬라이싱 원칙

이슈를 분해할 때 아래 기준을 따른다.

- **사용자 가치**: 각 이슈는 완료 시 사용자에게 관찰 가능한 가치를 전달해야 한다.
- **레이어 관통**: 이슈 하나가 UI · 로직 · 데이터 레이어를 수직으로 관통해야 한다.
- **독립 배포 가능**: 이슈 단독으로 완료·배포할 수 있어야 한다.
- **수평 슬라이싱 금지**: 특정 레이어(타입 정의, 훅 로직, UI 컴포넌트)만 건드리는 이슈는 만들지 않는다. 단, 후속 이슈들의 선행 조건이 되는 기반 작업(타입 확장, API 레이어 변경 등)은 예외로 허용하되 별도로 표기한다.

## issues.md 구조

### 파일 헤더

```
# {기능명} 기능 이슈 분해

## 수직 슬라이싱 원칙

각 이슈는 UI → 로직 → 데이터 레이어를 수직으로 관통해 사용자에게 측정 가능한 가치를 전달한다.
의존 순서: Issue 1 → ... (병렬 가능 이슈 명시)
```

### 각 이슈 구조

```
## Issue N: {제목}

**설명**
무엇을 구현하는지, 어떤 컴포넌트/훅/레이어가 관련되는지, 이전/다음 이슈와의 관계를 1–3문장으로 기술한다.

**완료 조건 (Acceptance Criteria)**
- 관찰 가능한 결과로 작성한다. 구현 세부사항이 아닌 외부에서 검증 가능한 상태로 표현한다.
- 각 항목은 독립적으로 테스트할 수 있어야 한다.

**시나리오 N — {시나리오 이름}**

*Given* 전제 조건 (현재 상태)
*When* 사용자 행동 또는 이벤트
*Then* 기대 결과 (관찰 가능한 변화)
```

## 이슈 분해 가이드

- PRD의 사용자 스토리(U1, U2, ...)를 출발점으로 삼는다.
- 연관된 스토리는 하나의 이슈로 묶어도 된다. (예: U1+U4+U6이 모두 "실시간 필터링"에 속하면 통합)
- 이슈 수는 기능 복잡도에 따라 조정하되, 3–6개가 적당하다.
- 각 이슈에 시나리오는 2–5개가 적당하다. 해피 패스와 엣지 케이스를 균형 있게 포함한다.
- 의존 순서와 병렬 가능 여부를 파일 헤더에 명시한다.

## 작성 규칙

- 모든 내용은 한국어로 작성한다.
- Given/When/Then의 주어는 항상 사용자 또는 시스템의 관찰 가능한 상태다.
- 완료 조건은 구현 방법이 아닌 결과로 작성한다. ("훅이 X를 반환한다"는 허용, "useEffect를 사용한다"는 금지)
- 저장 완료 후 파일 경로, 이슈 수, 커버된 사용자 스토리 목록을 사용자에게 보고한다.
