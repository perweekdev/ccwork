`$ARGUMENTS` 기능을 `docs/features/feature-planning-workflow.md`의 3단계 파이프라인에 따라 기획한다.

## 시작 전

1. `$ARGUMENTS`가 비어 있으면 기능명을 사용자에게 먼저 묻는다.
2. `docs/features/$ARGUMENTS/spec-original.md`를 읽는다.
   - 없으면 사용자에게 초기 아이디어를 직접 입력해달라고 요청한다.
3. 코드베이스에서 관련 파일을 탐색해 기존 구조·패턴을 파악한다.

---

## 단계 1: 요구사항 인터뷰 → spec-fixed.md

### 목적

spec-original.md의 모호성을 제거하고 엣지 케이스와 결정 사항을 확정한다.

### 인터뷰 방식

- **한 번에 하나씩** 질문한다. 질문마다 추천 방식과 이유를 함께 제시한다.
- 아래 8개 영역을 순서대로 다룬다. 이미 spec에 명확히 서술된 항목은 건너뛴다.

1. 이 기능의 primary user는 누구인가?
2. 최소 동작 시나리오 3개는? (가장 기본적인 사용 흐름)
3. 데이터 저장 방식 — 기존 구조와의 관계는?
4. 경계 조건 — 최대값, 빈 값, 중복, 동시성은?
5. 에러 처리 — 실패 시 사용자에게 어떻게 표시하는가?
6. 기존 UI 패턴과의 일관성 — 어떤 컴포넌트를 재사용할 수 있는가?
7. 성능 제약 — 허용 응답 시간, 데이터 크기 제한은?
8. 향후 확장 가능성 — 이 결정이 다음 feature에 영향을 주는가?

### 산출물

인터뷰 완료 후 `docs/features/$ARGUMENTS/spec-fixed.md`를 작성한다.

- 확정된 요구사항, 엣지 케이스 처리 방침, 임의 결정 항목을 포함한다.
- "용어 정의" 섹션에 Ubiquitous Language를 명시한다.

```
[GATE] spec-fixed.md를 사용자에게 보여주고, 확정 확인을 받을 때까지 단계 2로 넘어가지 않는다.
```

## 단계 2: PRD 작성 + 기술 결정(ADR) → prd.md

### 2-1. PRD 뼈대 생성

spec-fixed.md를 바탕으로 아래 구조의 PRD 초안을 작성한다. 기술 결정 섹션은 아직 TODO로 비워둔다.

- **개요** — 기능의 목적과 범위 (2–3문장)
- **사용자 스토리** — `~하는 사용자로서, ~하고 싶다` 형식, 인수 조건 포함
- **기술 결정** — (TODO: 2-2 이후 채움)
- **Out of Scope** — (TODO: 2-4에서 채움)
- **용어 정의** — spec-fixed.md의 Ubiquitous Language 동기화

### 2-2. 아키텍처 3안 비교 → 사용자 선택

코드베이스를 탐색해 최소 3가지 구현 안을 제안한다. 아래 **7가지 고정 기준**으로 비교표를 작성한다.

| #   | 기준                 | 설명                                           |
| --- | -------------------- | ---------------------------------------------- |
| 1   | 데이터 구조          | 어떤 형태로 데이터를 저장/표현하는가           |
| 2   | API 레이어 변경지점  | 기존 API에서 무엇이 바뀌는가                   |
| 3   | 상태관리 변경지점    | Context/Store에서 무엇이 추가되는가            |
| 4   | 핵심 동작            | 이 기능의 가장 중요한 동작이 어떻게 구현되는가 |
| 5   | 컴포넌트 구조        | 어떤 컴포넌트가 추가/수정되는가                |
| 6   | 기존 패턴과의 일관성 | 코드베이스의 기존 관습과 얼마나 맞는가         |
| 7   | 테스트 용이성        | 단위 테스트, E2E 테스트 작성이 쉬운가          |

```
[GATE] 비교표를 사용자에게 보여주고, 3안 중 하나를 선택받을 때까지 2-3으로 넘어가지 않는다.
```

### 2-3. 선택안을 ADR 4요소로 작성

사용자가 선택한 안을 PRD의 "기술 결정" 섹션에 ADR 형식으로 작성한다.

각 ADR은 아래 4요소를 반드시 포함한다. 요소 생략 금지.

```
**Context** — 이 결정이 필요한 맥락. 무엇을 풀고 있는가.
**Decision** — 무엇을 선택했는가.
**Alternatives** — 거부한 안과 각각의 거부 이유. (이유 없는 거부 금지)
**Consequences** — 트레이드오프. 장점뿐 아니라 단점도 반드시 명시.
```

### 2-4. Out of Scope 작성

"이번에 하지 않을 것"을 구체적으로 나열한다.

- spec에서 제외한 항목, 향후 확장 예정 항목을 포함한다.
- 각 항목은 소제목 + 1–2줄 이유로 작성한다.

```
[GATE] Out of Scope를 사용자에게 보여주고, 범위 확정 확인을 받을 때까지 prd.md를 저장하지 않는다.
```

### 산출물

확정 후 `docs/features/$ARGUMENTS/prd.md`를 저장한다.

## 단계 3: 이슈 분해 → issues.md

### 수직 슬라이싱 원칙

각 이슈를 분해할 때 아래 질문으로 검증한다.

> "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?"
>
> - **Yes** → 수직 슬라이스. 채택.
> - **No** → 다른 이슈와 합치거나 다시 나눈다.

수평 슬라이싱(레이어별: 타입만, 훅만, UI만) 금지.
이슈 크기는 반나절–하루 안에 TDD 사이클(Red→Green→Refactor) 완료 가능한 단위.

### issues.md 구조

`docs/features/tag/issues.md`를 포맷 기준으로 참조한다.

각 이슈는 아래 항목을 포함한다.

- **설명**: 무엇을 구현하는지, 관련 컴포넌트/레이어, 이전·다음 이슈와의 관계
- **완료 조건 (Acceptance Criteria)**: 관찰 가능한 결과로 작성. 구현 세부사항 금지.
- **시나리오 (Given-When-Then)**: 해피 패스와 엣지 케이스를 균형 있게 포함. 이슈당 2–5개.

파일 헤더에 의존 순서와 병렬 가능 여부를 명시한다.

### 산출물

`docs/features/$ARGUMENTS/issues.md`를 저장한다.

```
[GATE] issues.md를 사용자에게 보여주고, 수직 슬라이스 기준 충족 여부와 AC 구체성을 확인받는다.
```

### GitHub Issues 등록

issues.md가 확정되면 GitHub 이슈 등록 여부를 사용자에게 질문한다:

"issues.md의 이슈들을 GitHub에 등록할까요? (예 / 아니오)"

- **아니오**: 이 단계를 건너뛰고 단계 4로 진행한다.
- **예**: 각 이슈를 순서대로 등록한다. 이슈별로 생성할 제목과 본문을 미리 출력해 확인받은 뒤 생성한다.

  ```bash
  gh issue create --title "이슈 제목" --body "설명 + AC + 의존성"
  ```

  각 이슈 생성 후 URL을 출력한다.

## 단계 4: feature 브랜치 생성

issues.md 기반 TDD 사이클을 시작하려면 `feature/<spec>` 브랜치가 필요하다.

`$ARGUMENTS`를 소문자 + 하이픈으로 변환해 추천 브랜치명을 제안하고 사용자에게 확인을 받는다:

```
추천 브랜치명: feature/{feature-slug}
다른 이름을 원하면 직접 입력하세요. 생성하지 않으려면 "건너뛰기"를 입력하세요.
```

- **건너뛰기**: 이 단계를 생략하고 종료한다.
- **확인 또는 수정된 이름 입력**: 확정된 이름으로 브랜치를 생성한다.

  ```bash
  git checkout -b {확정된 브랜치명}
  ```

  브랜치 생성 후 아래 메시지를 출력한다:

  ```
  브랜치 {확정된 브랜치명} 생성 완료.
  이제 /tdd-loop <이슈 번호> 로 TDD 사이클을 시작할 수 있습니다.
  ```

## 게이트 요약

| 지점          | 확인 내용        | 통과 조건                             |
| ------------- | ---------------- | ------------------------------------- |
| 단계 1 후     | spec-fixed.md    | 모호성 제거, 용어 확정                |
| 단계 2-2 후   | 3안 비교표       | 사용자가 안을 선택함                  |
| 단계 2-4 후   | Out of Scope     | 범위 확정                             |
| 단계 3 후     | issues.md        | 수직 슬라이스, AC 구체성, 의존성 순서 |
| 단계 3 GitHub | GitHub 이슈 등록 | 사용자가 예/아니오를 선택함           |
| 단계 4        | feature 브랜치   | 사용자가 브랜치명을 확인/수정함       |

게이트에서 사용자 확인 없이 다음 단계로 진행하지 않는다.
