`$ARGUMENTS`에서 feature 이름과 이슈 번호를 파싱해, tdd-red가 작성한 실패 테스트를 통과시키는 최소 구현 코드를 작성한다.

## 시작 전

### 인자 파싱

`$ARGUMENTS`를 공백으로 분리해 첫 번째 토큰을 `{feature}`, 두 번째 토큰을 `{issue}`로 사용한다.

| 입력 예시             | feature  | issue |
| --------------------- | -------- | ----- |
| `/tdd-green tag 2`    | `tag`    | `2`   |
| `/tdd-green search 1` | `search` | `1`   |

누락된 값이 있으면 실행 전에 사용자에게 질문한다.

- `{feature}`가 없으면: "어떤 기능의 이슈인가요? (예: tag, search, ...)"
- `{issue}`가 없으면: "이슈 번호를 알려주세요."

두 값이 확정된 뒤 `docs/features/{feature}/issue-{issue}.md`를 읽어 시그니처와 시나리오 체크박스 목록을 파악한다.

파일이 없으면 즉시 멈추고 사용자에게 알린다 (`/tdd-red {feature} {issue}`를 먼저 실행해야 한다는 안내 포함).

---

## 단계 1: 실패 테스트 목록 확인

```bash
npm test
```

출력에서 `FAIL` 항목과 실패한 `it()` 이름 목록을 수집한다.

---

## 단계 2: collect 실패 감지

`docs/features/{feature}/issue-{issue}.md`의 체크박스(`- [ ]`) 총 건수와 vitest가 보고하는 `Tests` 수를 비교한다.

- **일치**: 다음 단계로 진행한다.
- **불일치**: import 대상 모듈이 없어 테스트가 인식되지 않는 것이다. 시그니처 섹션을 읽어 누락된 구현 파일에 대해 최소 스텁을 생성한 뒤 `npm test`를 다시 실행해 수를 맞춘다.

### 스텁 생성 규칙 (불일치 시에만)

기존 파일이 있으면 건드리지 않는다. 신규 파일만 생성한다.

| 반환 타입      | 스텁 반환값                |
| -------------- | -------------------------- |
| `string[]`     | `[]`                       |
| `string`       | `''`                       |
| `boolean`      | `false`                    |
| `Promise<T>`   | `Promise.resolve({} as T)` |
| React 컴포넌트 | `return <div />;`          |
| `void`         | _(반환 없음)_              |

---

## 단계 3: 디자인 컨텍스트 로드

구현 대상이 UI 컴포넌트인 경우에만 수행한다.

1. `docs/design-system/index.md`를 읽어 디자인 시스템 파일 구조를 파악한다.
2. 구현 대상 컴포넌트의 시각적 스펙(색상, 간격, 레이아웃, 상태별 표현)을 확인하기 위해 아래 파일을 읽는다:
   - `docs/design-system/components.md` — 컴포넌트별 Tailwind 클래스 패턴, Radius·Shadow 토큰
   - `docs/design-system/tokens.md` — CSS 변수 ↔ Tailwind 매핑, 색상 스케일
3. `docs/features/{feature}/issue-{issue}.md`에 "디자인 참고" 섹션이 있으면 함께 확인한다.
4. CLAUDE.md의 스타일링 컨벤션(`docs/design-system/rules.md` 포함)도 참조한다.

디자인 문서가 없으면 이 단계를 건너뛰고 CLAUDE.md 컨벤션만 따른다.

---

## 단계 4~7: 테스트별 구현 루프

첫 번째 실패 테스트부터 시작해 아래 루프를 반복한다.

### 4. 최소 구현 작성

- 해당 테스트를 통과시키는 코드만 작성한다. 다른 테스트를 위한 코드는 쓰지 않는다.
- UI 컴포넌트는 테스트가 요구하는 동작을 충족하면서 단계 3에서 확인한 디자인 스펙을 함께 적용한다.
  - `docs/design-system/components.md`에 명시된 Tailwind 클래스와 레이아웃 구조를 그대로 사용한다.
  - 디자인 문서에 없는 시각적 요소는 임의로 추가하지 않는다.

### 5. 피드백 루프

```bash
npm test
```

- 해당 테스트가 통과하고 기존 테스트에 회귀가 없으면: 단계 6으로 이동한다.
- 해당 테스트가 아직 실패 중이면: 에러 메시지를 분석해 코드를 수정한 뒤 5번을 다시 실행한다.
- **최대 5회 반복 후에도 실패하면**: 실패 원인을 사용자에게 보고하고 중단한다.
- 기존에 통과하던 테스트가 새로 실패하면: 즉시 사용자에게 보고하고 회귀 원인을 해결한다.

### 6. 체크박스 업데이트

`docs/features/{feature}/issue-{issue}.md`에서 해당 시나리오의 `- [ ]`를 `- [x]`로 변경한다.

### 7. 다음 테스트로 이동

다음 실패 테스트로 이동해 4번부터 반복한다. 모든 테스트가 통과할 때까지 계속한다.

---

## 단계 8: 커버리지 측정

전체 테스트가 통과한 뒤 실행한다.

```bash
npx vitest run --coverage
```

- 미커버 라인을 확인하고, 시나리오에서 누락된 케이스인지 사용자에게 보고한다.
- 커버리지를 높이기 위한 테스트 추가는 금지한다. 참고용으로만 활용한다.

---

## 단계 9: 결과 요약

- 구현된 파일 목록 및 경로
- 시나리오별 통과 여부
- `npm test` 최종 결과: 통과한 테스트 수 / 전체 테스트 수
- 커버리지 미달 라인 목록 (있을 경우)

---

## 제약

- 테스트 파일(`*.test.ts`, `*.test.tsx`) 수정 금지.
- 테스트에 없는 기능 구현 금지.
- 불필요한 추상화 금지.
- 디자인 문서에 없는 시각적 요소 임의 추가 금지.
