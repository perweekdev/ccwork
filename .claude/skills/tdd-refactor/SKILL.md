description: 테스트 통과 상태를 유지하면서 이번 이슈에서 변경된 코드를 리팩토링한다. tdd-green 이후 Refactor 단계 시작 시 사용한다.
argument-hint: <이슈 번호>
allowed-tools: Read Write Bash Edit Glob Grep
---

`$ARGUMENTS`에서 이슈 번호를 파싱해, tdd-green이 통과시킨 테스트를 깨뜨리지 않으면서 코드 구조를 개선한다.

## 시작 전

### 인자 파싱

`$ARGUMENTS`를 이슈 번호(`{issue}`)로 사용한다.

누락된 값이 있으면 실행 전에 사용자에게 질문한다.

- `{issue}`가 없으면: "리팩토링할 이슈 번호를 알려주세요."

---

## 단계 1: 프로젝트 컨벤션 파악

`CLAUDE.md`를 읽어 아래 항목을 파악한다:

- 네이밍 규칙 (파일명, 함수명, 이벤트 핸들러, 불리언 state/props 접두사)
- 컴포넌트 구현 패턴 (named export, Props 타입 위치, Context 접근 계층)
- 상태 관리 원칙 (상태 종류별 위치 기준)
- API 호출 패턴 (fetch 직접 사용, res.ok 검사, 타임스탬프 생성 위치)
- 스타일링 규칙 (`docs/design-system/rules.md` 포함)

---

## 단계 2: 전체 테스트 통과 확인

```bash
npm test
```

테스트가 전부 통과하지 않으면 즉시 중단하고 사용자에게 알린다. Refactor 단계는 Green 상태에서만 시작한다.

---

## 단계 3: 리팩토링 대상 파일 식별

### 3-1. 변경 파일 추출

```bash
git diff main...HEAD --name-only -- src/
```

### 3-2. 테스트 파일 제외

`*.test.ts` / `*.test.tsx` 패턴에 해당하는 파일을 목록에서 제거한다.

### 3-3. 범용 점검 기준으로 검토

남은 파일 각각에 대해 아래 다섯 가지 기준을 적용한다:

| 기준                        | 점검 내용                                                                 |
| --------------------------- | ------------------------------------------------------------------------- |
| **중복 제거**               | 같은 로직이 여러 곳에 반복되는가                                          |
| **네이밍 명확성**           | 함수·변수·컴포넌트 이름이 의도를 드러내는가                               |
| **단일 책임**               | 하나의 함수/컴포넌트가 너무 많은 일을 하는가                              |
| **불필요한 복잡도**         | 매직 넘버, 과도한 중첩, 불필요한 추상화가 있는가                          |
| **CLAUDE.md 컨벤션 불일치** | 프로젝트 패턴(네이밍, export 방식, 상태 위치 등)과 어긋나는 코드가 있는가 |

---

## 단계 4: 리팩토링 목록 보고 → 승인 대기

점검 결과를 아래 형식으로 정리해 사용자에게 보고한다:

```
파일: src/...
- [기준] 문제 설명 → 제안 변경 내용
```

개선 대상이 없으면 "리팩토링이 필요한 부분을 발견하지 못했습니다."를 출력하고 종료한다.

**사용자 승인을 받은 뒤에만 다음 단계로 진행한다.**

---

## 단계 5~6: 변경 → 테스트 → 반복

승인된 항목을 하나씩 처리한다.

### 5. 변경 적용

승인된 항목 하나에 대해서만 코드를 수정한다.

### 6. 테스트 확인

```bash
npm test
```

- **전체 통과**: 다음 항목으로 이동한다.
- **실패**: 즉시 변경을 되돌리고(`git restore` 또는 Edit 역적용), 사용자에게 실패 원인을 보고한 뒤 해당 항목을 건너뛴다.

모든 승인 항목을 처리할 때까지 반복한다.

---

## 단계 7: 리팩토링 요약

처리 결과를 아래 형식으로 출력한다:

```
변경 파일: src/...
- 적용한 변경: ...
- 건너뛴 항목 (테스트 실패): ... (해당 시)
npm test: {통과 수} / {전체 수}
```

---

## 제약

- 새 기능 추가 금지.
- 테스트 파일(`*.test.ts`, `*.test.tsx`) 수정 금지.
- `src/` 디렉토리 밖 파일 수정 금지.
- `git diff main...HEAD --name-only -- src/` 결과에 포함된 파일만 수정한다.
- 테스트가 깨지면 즉시 롤백하며 변경을 강행하지 않는다.
