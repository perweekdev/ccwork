# 태그 기능 PRD

## 개요

노트에 태그를 붙여 분류할 수 있게 한다. 사용자는 NoteEditor에서 태그를 자유롭게 추가·삭제하고, 노트 저장 시 title/content와 함께 일괄 반영한다. 사이드바 NoteItem에도 태그가 소형 뱃지로 노출되어 목록에서 태그를 한눈에 확인할 수 있다.

---

## 사용자 스토리

| #   | 스토리                                                                                             | 인수 조건                                                          |
| --- | -------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| U1  | 노트를 편집하는 사용자로서, 태그를 입력해 노트를 분류하고 싶다.                                    | Enter 또는 `,` 입력 시 태그가 뱃지로 확정된다.                     |
| U2  | 노트를 편집하는 사용자로서, 잘못 추가한 태그를 제거하고 싶다.                                      | 각 뱃지의 `×` 버튼 클릭 시 해당 태그가 즉시 제거된다.              |
| U3  | 노트를 편집하는 사용자로서, 저장 버튼 하나로 제목·본문·태그를 한 번에 저장하고 싶다.               | 저장 버튼 클릭 시 tags 배열이 title/content와 함께 API로 전송된다. |
| U4  | 노트 목록을 훑는 사용자로서, 사이드바에서 각 노트의 태그를 미리 확인하고 싶다.                     | NoteItem에 해당 노트의 tags가 소형 뱃지로 표시된다.                |
| U5  | 노트를 편집하는 사용자로서, 입력 실수(공백만 입력, 중복, 너무 긴 태그)가 자동으로 걸러지길 원한다. | 공백 전용·20자 초과·normalize 후 중복 태그는 조용히 무시된다.      |
| U6  | 노트를 편집하는 사용자로서, 새 노트를 생성할 때 태그가 없는 상태에서 시작하길 원한다.              | 새 노트의 tags는 `[]`로 초기화된다.                                |

## 기술 결정 (ADR)

### ADR-1. 태그 정규화(normalize) 방식

- **Context**: 스펙은 소문자 변환, 앞뒤 trim, 내부 공백 제거를 요구한다. 이 로직을 어느 레이어에 두느냐에 따라 테스트 범위와 호출 지점이 달라진다.
- **Decision**: normalize 로직을 `useTagInput` 훅 내부의 `addTag()` 함수 안에 캡슐화한다. 호출자(`TagInput` 컴포넌트)는 raw 입력값을 그대로 전달하고, 훅이 내부적으로 normalize → 중복 검사 → 추가 순서로 처리한다.
- **Alternatives**: `utils/tag.ts`에 순수 함수로 분리해 훅과 테스트 양쪽에서 import하는 방식. 훅 자체가 단위 테스트 가능하므로 추가 분리의 이득이 크지 않아 채택하지 않았다.
- **Consequences**:
  - (장점) normalize 규칙 변경 시 `useTagInput` 한 곳만 수정하면 된다. 훅을 단독으로 테스트하면 normalize 동작을 컴포넌트 없이 검증할 수 있다.
  - (단점) normalize 로직이 훅 내부에 캡슐화되어 있어, 훅을 거치지 않고 순수 함수로만 재사용하려면 별도 추출이 필요하다.

### ADR-2. 태그 입력 컴포넌트 분리 여부

- **Context**: 태그 추가·삭제·normalize·중복 검사 로직을 `NoteEditor` 내부에 인라인으로 작성하면 변경 범위는 최소지만, 로직이 컴포넌트에 묻혀 단위 테스트가 어려워진다.
- **Decision**: `useTagInput` 커스텀 훅으로 상태·로직을 분리하고, `TagInput` 컴포넌트는 훅이 반환한 값만 props로 받아 렌더링만 담당한다. `NoteEditor`는 `useTagInput()`을 호출해 `<TagInput>`에 내려준다.
- **Alternatives**: 인라인 확장(안 1) — 파일 추가 없이 `NoteEditor`에 직접 작성. 로직 단독 테스트가 불가능해 채택하지 않았다.
- **Consequences**:
  - (장점) normalize·중복 검사 로직을 컴포넌트 마운트 없이 단위 테스트할 수 있다. `TagInput`은 `NoteItem`과 같은 순수 props 컴포넌트 패턴을 따르므로 기존 아키텍처 계층 원칙과 일치한다.
  - (단점) 파일이 `useTagInput.ts`와 `TagInput.tsx` 두 개 추가되고, `NoteEditor` → `useTagInput` → `TagInput`으로 props 전달 계층이 한 단계 늘어난다.

### ADR-3. 저장 전 로컬 태그 상태 관리 위치

- **Context**: 태그 변경은 저장 버튼 클릭 전까지 서버에 반영되지 않아야 한다. 상태를 Context에 두면 즉시 전파되고, `NoteEditor` 로컬에 두면 저장 전까지 격리된다.
- **Decision**: `useTagInput` 훅이 반환하는 `tags` state를 `NoteEditor` 로컬에서 관리한다. 저장 버튼 클릭 시 `title`, `content`와 함께 Context의 `updateNote` / `createNote`에 일괄 전달한다.
- **Alternatives**: Context에 `pendingTags` 상태를 두는 방식 — 편집 중 태그가 전역으로 노출되어 다른 컴포넌트에 부작용을 줄 수 있어 채택하지 않았다.
- **Consequences**:
  - (장점) 저장하지 않고 다른 노트로 이동하면 태그 변경이 자연스럽게 버려진다(스펙 요구사항 충족). Context는 서버와 동기화된 데이터만 보유한다는 기존 원칙을 유지한다.
  - (단점) 저장 없이 이동하면 변경이 소실되는 동작이 사용자에게 명시적으로 안내되지 않으면 의도치 않은 데이터 손실로 느껴질 수 있다.

### ADR-4. NoteItem 태그 표시 데이터 흐름

- **Context**: `NoteItem`은 이미 `note: Note` 객체 전체를 prop으로 받는다. `tags`를 별도 prop으로 추가하는 방법과 `note.tags`를 직접 읽는 방법이 있다.
- **Decision**: `NoteItemProps`를 변경하지 않고 기존 `note` prop에서 `note.tags`를 직접 접근해 렌더링한다.
- **Alternatives**: `tags: string[]`을 별도 prop으로 추가 — `Note` 타입과 props가 이중으로 관리되어 필드 추가 시 양쪽을 모두 수정해야 하므로 채택하지 않았다.
- **Consequences**:
  - (장점) `NoteList`에서 `NoteItem`을 렌더링하는 호출부 변경이 없다. `Note` 타입에 필드가 추가되어도 `NoteItem` props 시그니처는 그대로 유지된다.
  - (단점) `NoteItem`이 실제로 사용하지 않는 `content`, `createdAt` 등 `Note`의 모든 필드에 암묵적으로 의존하게 되어, 컴포넌트만 보고 실제 사용 데이터를 파악하기 어렵다.

## Out of Scope

### 태그 기반 필터링 / 검색

- 태그를 클릭하거나 검색창에 태그를 입력해 노트 목록을 좁히는 기능은 이번 스펙 범위 밖이다.
- 향후 사이드바에 태그 필터 패널을 별도 구현할 때 포함한다.

### 자동완성 / 태그 제안

- 기존 태그 목록을 기반으로 입력 중 후보를 드롭다운으로 보여주는 기능은 제외한다.
- 전체 태그 집합을 집계하는 API 또는 프론트엔드 파생 상태가 필요하므로 별도 스펙에서 설계한다.

### 태그 전역 관리 (rename / delete)

- 특정 태그를 전체 노트에서 일괄 이름 변경하거나 삭제하는 관리 화면은 제외한다.

### 태그 색상 / 아이콘 커스터마이징

- 태그별 색상 지정, 아이콘 연결 등 시각적 커스터마이징은 제외한다. 모든 뱃지는 단일 스타일을 공유한다.

### 태그 순서 변경 (드래그 앤 드롭)

- 뱃지를 드래그해 순서를 바꾸는 인터랙션은 제외한다. 태그 순서는 추가된 순서를 유지한다.

### 태그 공유 / 협업

- 노트 공유, 멀티유저 환경에서의 태그 충돌 처리 등은 현 단일 사용자 구조에서 고려하지 않는다.

### 태그 입력 중 자동 저장

- 태그가 추가·삭제될 때마다 즉시 API를 호출하는 자동 저장은 제외한다. 저장은 항상 명시적인 저장 버튼 클릭으로만 이루어진다.

### 태그 최대 개수 제한

- 현 스펙은 태그 개수 상한을 두지 않는다. 향후 UX 검토 후 필요 시 추가한다.

## 용어 정의

| 용어                   | 정의                                                                                                        |
| ---------------------- | ----------------------------------------------------------------------------------------------------------- |
| **태그(tag)**          | 노트에 붙이는 짧은 문자열 레이블. Note 객체의 `tags: string[]` 배열 원소 하나.                              |
| **뱃지(badge / chip)** | 확정된 태그를 시각적으로 나타내는 UI 요소. NoteEditor에서는 일반 크기, NoteItem에서는 소형으로 표시된다.    |
| **미확정 입력값**      | 사용자가 입력 중이나 아직 Enter/`,`로 확정하지 않은 텍스트. 다른 노트로 이동하면 버려진다.                  |
| **normalize**          | 태그 문자열에 적용하는 전처리 규칙: 소문자 변환 + 앞뒤 trim + 내부 공백 제거. 중복 비교 전에 항상 수행한다. |
| **일괄 저장**          | 저장 버튼 클릭 한 번으로 title, content, tags를 하나의 PUT 요청으로 전송하는 방식.                          |
