# 검색 기능 PRD

## 개요

사이드바 상단 검색창에 키워드를 입력하면 노트 목록이 실시간으로 필터링된다. 검색 범위는 노트 제목과 내용 전체이며, 서버 요청 없이 클라이언트 메모리의 `notes[]`를 직접 걸러낸다. 검색어는 `NoteList` 로컬 상태로 관리되고, 새 노트 생성 시 초기화된다.

## 기술 결정 (ADR)

### ADR-1. 검색어 상태 위치

- **Context**: 검색어 상태를 `NoteList` 로컬에 두면 범위가 최소로 유지되지만, 다른 컴포넌트(예: 새 노트 버튼이 있는 `App.tsx`)에서 검색어를 초기화해야 할 때 props drilling이 발생할 수 있다.
- **Decision**: `searchQuery` state를 `NoteList` 로컬에서 관리한다. 새 노트 생성 시 초기화는 `NoteList` 내부에서 `onNewNote` 핸들러를 래핑해 처리한다.
- **Alternatives**: `App.tsx`로 끌어올려 `NoteList`에 prop으로 내리는 방식 — `selectedNoteId`, `isCreating`과 함께 UI 상태 집중화가 가능하지만, 검색어가 다른 컴포넌트에 불필요하게 노출되어 채택하지 않았다.
- **Consequences**:
  - (장점) 검색은 NoteList 뷰 전용 UI 상태이므로 해당 컴포넌트 안에서 완결된다. 상태 테이블(`App.tsx` — 전역 UI 선택 상태 / `NoteList` — 필터 상태)의 기존 원칙과 일치한다.
  - (단점) 향후 외부 컴포넌트에서 검색어를 읽거나 초기화해야 하는 요구가 생기면 끌어올리기가 필요하다.

### ADR-2. 필터링 방식 — 클라이언트 vs 서버

- **Context**: json-server는 `GET /notes?q=keyword` 쿼리를 지원하지만 전문 검색 인덱스가 없다. Context의 `notes[]`는 이미 메모리에 올라와 있다.
- **Decision**: 서버 요청 없이 Context의 `notes[]`를 클라이언트에서 직접 필터링한다.
- **Alternatives**: 서버 쿼리 파라미터 방식 — 검색마다 API 호출이 발생하고 디바운스·로딩 상태 처리가 추가로 필요해 채택하지 않았다.
- **Consequences**:
  - (장점) API 호출이 없으므로 네트워크 지연 없이 입력 즉시 반응한다. 디바운스, 로딩 스피너, 에러 처리가 불필요하다.
  - (단점) notes 배열 전체가 메모리에 있어야 한다. 노트 수가 수만 건 이상으로 늘어나면 클라이언트 필터 성능이 저하될 수 있다(현 단계에서는 무관).

### ADR-3. 검색 범위 — 제목만 vs 제목+내용

- **Context**: 제목만 검색하면 노이즈가 적지만, 사용자가 내용 키워드로 노트를 찾는 경우를 커버하지 못한다.
- **Decision**: 제목(`title`)과 내용(`content`) 모두를 검색 범위에 포함한다.
- **Alternatives**: 제목만 검색 — 구현이 단순하지만 내용 기반 탐색이 불가능해 채택하지 않았다.
- **Consequences**:
  - (장점) 사용자가 제목을 기억하지 못해도 내용 키워드로 노트를 찾을 수 있다.
  - (단점) 내용이 긴 노트가 많아질수록 의도치 않은 노트가 결과에 포함될 수 있다. 향후 제목 우선 정렬로 보완 가능하다.

### ADR-4. 디바운스 적용 여부

- **Context**: 입력마다 필터링이 트리거된다. 서버 요청이라면 디바운스가 필수지만, 클라이언트 연산은 비용이 매우 작다.
- **Decision**: 디바운스를 적용하지 않는다. 입력 이벤트마다 즉시 필터링한다.
- **Alternatives**: 150–300ms 디바운스 — 서버 호출이 없는 지금은 입력 지연만 추가되어 채택하지 않았다.
- **Consequences**:
  - (장점) 입력 즉시 반응하므로 UX가 자연스럽다. `useCallback`·`useMemo` 같은 최적화 장치 없이도 현 데이터 규모에서는 무방하다.
  - (단점) 노트 수가 수천 건 이상이 되거나 검색 연산이 복잡해지면 별도 최적화가 필요하다.

## Out of Scope

### 검색어 하이라이트

- 검색 결과에서 일치하는 키워드를 강조 표시하는 기능은 이번 스펙 범위 밖이다.
- 별도 하이라이트 유틸리티(예: `mark.js`) 또는 커스텀 렌더링 로직이 필요하므로 별도 스펙에서 설계한다.

### 태그 검색 연동

- 태그 이름으로 노트를 필터링하는 기능은 제외한다.
- 태그 기능 구현 후 필터 범위 확장 스펙으로 별도 다룬다.

### 서버 측 전문 검색

- Elasticsearch, Meilisearch 등 전문 검색 엔진 연동은 제외한다.
- 현 json-server 기반 구조에서는 클라이언트 필터링으로 충분하다.

### 검색 히스토리 / 최근 검색어

- 이전 검색어를 저장해 재사용하는 기능은 제외한다.

### 검색어 유지 (페이지 새로고침)

- 새로고침 후에도 검색어가 유지되도록 URL 파라미터나 localStorage에 저장하는 기능은 제외한다.

## 용어 정의

| 용어                    | 정의                                                                                                               |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------ |
| **검색어(searchQuery)** | 사용자가 입력한 필터링 기준 문자열. `NoteList` 로컬 state로 관리되며, 새 노트 생성 시 빈 문자열로 초기화된다.      |
| **클라이언트 필터링**   | 서버 요청 없이 메모리 내 `notes[]` 배열을 검색어로 걸러내는 동작.                                                  |
| **검색 범위**           | 필터링 대상 필드. 노트의 `title`과 `content` 모두 포함한다.                                                        |
| **normalize**           | 검색어와 노트 데이터에 공통으로 적용하는 전처리 규칙: 소문자 변환 + 앞뒤 trim. 대소문자 무관 비교를 위해 수행한다. |
| **검색 결과 없음 상태** | 필터링 결과가 빈 배열일 때 NoteList 목록 자리에 표시되는 안내 메시지 상태.                                         |
